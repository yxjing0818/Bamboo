# A 轨专属：成熟几何工程师的 12 周 AI 提效计划

> 基于 `几何AI学习_v0_原始12周计划.md` 重构。本版**只保留 A 轨**——把 AI 当作生产力工具用，而不是去造 AI 系统。
> **不包含**：Anthropic/OpenAI SDK 编程、MCP server、RAG 工程、PyTorch/几何深度学习。
> **包含**：精准 Prompt、上下文管理、IDE 集成、Code Review、SymPy + AI 推导、失败模式识别。

## 适用人群与目标声明

**你是**：
- 熟练的几何造型算法工程师，B 样条 / NURBS / 网格 / 拓扑 / OCCT 这些不需要从头讲
- C++ 老兵，对 Eigen、模板、内存有直觉
- AI 几乎零基础，没系统用过 Prompt，也没让 AI 帮你写过工业代码

**你不想要**：
- 学怎么调 SDK 写 AI 应用
- 学神经网络、PointNet、SDF 那些科研型内容
- 把工作流改成"AI 中心化"

**你想要**：
- **每天少花 2-3 小时在重复劳动**（公式推导、boilerplate 代码、文档检索、debug 思路探索）
- 一套**可重用的 Prompt 模板**，每次新任务不用从零想
- **判断力**：什么时候用 AI，什么时候压根不该用

# 阶段一：表达精度（第 1-4 周）

**核心命题**：你已经知道几何算法该长什么样，但 AI 不知道。**你的任务是让 AI 精确知道你想要什么**。

## 第 1 周：公式化 Prompt vs 大白话

**周目标**：建立"prompt 质量直接决定代码质量"的直觉。沉淀 3 个可重用模板。

### 周一：先做对照实验（2h）

用 Claude.ai 网页版，对同一个任务发**三种 prompt**：

| 版本       | Prompt                                                            | 期望观察                       |
| -------- | ----------------------------------------------------------------- | -------------------------- |
| A 大白话    | "用 C++ 写一个三阶 Bezier 求值函数"                                         | 多半返回 Bernstein 展开，可能没有边界处理 |
|          |                                                                   |                            |
| B 加公式    | 同上 + 写出 $B(t) = \sum_{i=0}^{3}\binom{3}{i}(1-t)^{3-i}t^i P_i$     | 仍可能用展开式                    |
| C 加算法+约束 | 同上 + 指定 De Casteljau + 假设 Point3D 已有运算符重载 + 要求处理 $t \notin [0,1]$ | 应该给出工业级代码                  |

把三段代码并排放，在你的 `notes_week01.md` 里写下：
- 三段在数值稳定性上的差异（哪段会在退化输入下崩）
- 三段在风格一致性上的差异
- **如果你团队新人写出 A 版代码，你会让他重写吗？**

### 周二：B 样条基函数的"思维链" Prompt（2h）

任务：B 样条基函数（Cox-de Boor）。

错误做法：直接说"写 Cox-de Boor 的 C++ 实现"。

正确做法：
```
请按以下顺序回答，不要跳步：
1. 写出 N_{i,p}(u) 的递推公式（带边界条件）
2. 列出递推过程中所有可能除零的位置，给出 epsilon 处理策略
3. 在递推到 p=0 时如何处理半开区间 [u_i, u_{i+1})（注意右端点的特殊情况）
4. 在以上分析基础上，给出 C++ 实现，函数签名：
   double basis(int i, int p, double u, const std::vector<double>& knots);
```

**关注点**：要求 AI **先分析后写代码**。这是整 12 周里最重要的 prompt 技巧——叫 **Chain-of-Thought 引导**。

### 周三：NURBS 的奇点声明（2h）

任务：NURBS 曲线求值。

模板要素：
```
在生成代码前，请先列出 NURBS 求值的所有奇异情况：
- 权重为零或负
- 节点重复度等于阶数（端点重合）
- 浮点累加导致的归一化误差
对每个奇异情况说明：检测方法、处理策略（exception / fallback / 静默 epsilon）。
然后再写代码。
```

**理解**：这种"先列异常清单"的 prompt，是把你脑子里的隐性知识**强制显性化**，让 AI 不漏处理。

### 周四：把三个 prompt 抽成模板（2h）

新建 `prompt_library/` 目录。提炼这周的核心 prompt 套路：

`prompt_library/01_geometric_algorithm.md`：
```markdown
## 用途：让 AI 写工业级几何算法 C++ 代码

## 模板
你是工业级几何算法工程师。请按以下顺序回答：
1. 写出 {算法名} 的数学定义（用 LaTeX）
2. 列出所有可能的退化/奇异输入（不少于 {N} 种）
3. 对每个退化情况给出处理策略
4. 给出 C++ 实现，要求：
   - 使用 {算法名选项}（避免使用 {数值不稳定的替代}）
   - 函数签名：{签名}
   - 假设 {可用的项目类型，如 Point3D 重载} 已存在
5. 给出至少 3 个能触发退化分支的单元测试输入

## 填坑变量
{算法名}, {N}, {算法名选项}, {数值不稳定的替代}, {签名}
```

### 周五：把模板真正用一次（2h）

用上面的模板，对一个**你以前自己写过的几何算法**重新发起 prompt（比如点到 NURBS 曲面投影、网格半边迭代器等）。

**对比 AI 写的版本和你原版**，记录在 `notes_week01.md`：
- AI 漏掉了哪些你考虑过的退化？
- AI 想到了哪些你没想过的？
- AI 在哪些地方过度防御（加了你认为多余的检查）？

### 周末：失败日志（1h）

新建 `failure_log.md`。本周至少记 3 条 AI 翻车的样本，格式：

```markdown
## 2026-XX-XX [任务名]
**症状**：AI 返回了 X
**根因**：我的 Prompt 没说明 Y
**修正**：在模板里加入了 Z
```

**失败日志比成功日志值钱十倍**。坚持 12 周写。

## 第 2 周：退化输入的防御性提示词

**周目标**：训练 AI 写出能扛恶意输入的代码。沉淀"退化检查清单" prompt 模板。

### 周一-周二：点到 B 样条曲线投影（Newton-Raphson）（4h）

模板要点：
```
任务：实现点 P 到 B 样条曲线 C(u) 的最近点求解（point inversion）。

请按以下逻辑生成代码：
1. 写出目标函数 f(u) = (C(u) - P) · C'(u) 及其导数
2. 用 Newton-Raphson 迭代求 f(u) = 0
3. 必须包含的退化处理：
   - 初值 u_0 选择策略（推荐：先对 control polygon 做粗粒度采样取最近）
   - 迭代步长过小时（|Δu| < 1e-6）的早停
   - 迭代发散（|f| 不下降连续 3 次）时的 fallback：bounding box 二分细分
   - u 跑出参数域时的 clamp 策略（开/闭曲线区分）
4. 容差参数：tol_distance, tol_parameter, max_iter 全部作为可配置入参，给出建议默认值
5. 函数签名：{你的项目签名}
```

**关键技巧**：你已经知道的"工业实务"（比如先粗采样取初值），**必须自己写进 prompt**。AI 不会自动想到。

### 周三-周四：点在多边形内（射线法）（4h）

这是经典退化重灾区。模板：
```
任务：判断点 P 是否在 2D 多边形（顶点序列 V）内部。用射线法。

请在生成代码前，明确处理以下情况：
1. 射线穿过多边形顶点 → 用什么 tie-breaking 规则（推荐：只算"向上"的边）
2. 射线与多边形边共线 → 跳过该边的判定
3. 多边形自相交 / 非简单多边形 → 行为约定（even-odd / winding number？）
4. 点恰好在边上 → 返回 true / false / 单独枚举值
5. 多边形顶点数 < 3 → 提前返回 false
对每条规则，在代码注释里写明"为什么这么处理"。
```

**亲手测试**：拿你的代码库里**最病态的多边形数据**（如果有的话），扔进 AI 生成的代码看结果。

### 周五：Möller-Trumbore 三角形相交（2h）

这次反着用 AI——**让它教你算法**，而不是写代码：
```
请详细解释 Möller-Trumbore 算法的几何推导：
1. 为什么不显式构造平面方程也能判定相交
2. 为什么算法不需要预归一化的法向量
3. 算法中哪一步的行列式如果为零意味着什么几何状态
4. 与基于平面方程的传统方法相比，数值上为什么更稳定
```

读懂后再让它写代码。**这是 AI 最高价值的场景**：你已经能验证它说的对不对，所以可以放心借它的脑子加速学习。

### 周末：抽出"退化检查清单"模板（2h）

`prompt_library/02_degenerate_input_checklist.md`：
```markdown
## 用途：让 AI 生成几何代码前，先穷举退化输入

## 模板
在为 {算法名} 生成 C++ 代码之前，请先生成一份退化输入清单，按以下分类穷举：

1. **拓扑退化**：输入元素的拓扑关系塌缩（共点、共线、共面、自相交）
2. **数值退化**：浮点极小量、量级悬殊、NaN/Inf、subnormal
3. **参数域退化**：参数越界、闭合 vs 开放、重复度等
4. **配置退化**：算法假设不满足（如非凸、非简单、非流形）

对每条列：触发条件 → 检测方法 → 处理策略 → 验证用例。

然后再生成代码，在每条退化处理处带行内注释 `// degen: <分类编号>`。
```

更新 `failure_log.md`。

## 第 3 周：上下文注入——让 AI 用你的代码风格

**周目标**：解决"AI 总用 std::vector 不用我的 ArrayN"这种风格污染问题。

### 周一-周二：自定义类型注入（4h）

模板：
```
以下是项目的核心几何类型头文件，请严格使用其中定义的类和运算符，
**禁止引入任何 STL 容器**，**禁止使用 std::sqrt 之外的 cmath**，
**禁止 new/delete 显式分配**。

[贴 Point3D.h 全文]
[贴 Vector3D.h 全文]
[贴 Matrix3x3.h 全文]

任务：实现 {功能}，函数签名：{签名}。
```

**实务技巧**：
- 把头文件全文贴进去，**比简化描述效果好 10 倍**
- 一次会话尽量做完所有相关功能，避免重复贴大段头文件
- 单次会话超过 30k tokens 后效果会下降，**及时开新会话**

### 周三-周四：Eigen 深度调用（4h）

任务：基于点云的最小二乘平面拟合。

模板：
```
用 Eigen 实现 N 个三维点的最小二乘平面拟合。

约束：
1. 输入是 const double* points（row-major，N×3），通过 Eigen::Map 处理，避免内存拷贝
2. 使用 JacobiSVD 求法向量（最小奇异值对应的右奇异向量）
3. 输出：平面法向 normal (Vector3d) + 一个平面上的点 centroid (Vector3d)
4. 退化处理：
   - N < 3 → throw 或返回 std::optional<...> 的空值
   - 所有点共线 → 检测最小奇异值与次小奇异值之比 < epsilon → 返回失败
5. 性能要求：避免 N×3 矩阵的显式临时变量，**直接对 (points - centroid).adjoint() * (points - centroid)** 这样的 3×3 协方差矩阵做 SVD

给出代码并解释每个性能关键点的原因。
```

**学到的**：和 Eigen 配合时，AI 经常给出"能跑但低效"的代码（多次临时拷贝）。**你的价值是认得出这种低效**。

### 周五：Code Review（2h）

把你自己写过的一段复杂几何代码（推荐：网格高斯曲率、半边遍历类的东西）发给 AI：

模板：
```
请以 C++ 性能专家身份 review 以下代码。重点关注：
1. 堆内存分配（new/delete、std::vector 重分配）
2. 缓存命中（连续访问 vs 跳跃访问）
3. 重复计算（同一表达式在循环内多次出现）
4. 编译器无法优化的写法（aliasing、虚函数调用）
5. 算法层面是否有更好的复杂度

对每个 issue：行号 + 问题描述 + 严重程度（critical/major/minor）+ 改进方案。
不要进行风格层面的批评（命名、缩进）。
```

通常 AI 能找出 2-3 个你忽视的优化点。**接受率应该在 30-50% 之间**——太低说明 prompt 不准，太高说明你之前代码太糙。

## 第 5 周：SymPy + AI 推导梯度

**周目标**：告别手算偏导数，且能验证 AI 推导没出错。

### 周一-周三：让 AI 写 SymPy 脚本（6h）

模板：
```
我需要推导以下能量泛函关于变量 {x_i} 的梯度：

E({x_i}) = [写出 LaTeX 公式]

请生成一段 Python（SymPy）脚本：
1. 声明所有符号变量（包括常数和待求变量）
2. 写出 E 的 SymPy 表达式
3. 求 dE/dx_i 的解析式，对每个 i
4. 用 simplify() 化简
5. 打印 LaTeX 格式
6. 用 sympy.cse() 提取公共子表达式
7. 用 sympy.ccode() 输出 C++ 代码

不要直接给我手推的公式——我要的是**脚本，不是结果**。脚本跑出来的结果我才信。
```

**关键点**：**永远让 AI 写脚本**，**永远跑一遍验证**。AI 自己手推公式 70% 会在符号上漏个负号。

### 周四-周日：Hessian 矩阵生成（4h）

对曲面拟合的能量函数，让 AI 给出 SymPy 脚本求 Hessian。

技巧：
- 用 `sympy.hessian(E, [x1, x2, ...])`
- 用 `sympy.MatrixSymbol` 处理矩阵化变量
- 用 `from sympy.utilities.codegen import codegen` 自动生成 `.cpp` + `.h`

模板：
```
生成 SymPy 脚本完成以下流程：
1. 定义符号
2. 定义能量 E
3. 计算 Hessian H = sympy.hessian(E, vars)
4. 对 H 的每个非零元素 H[i,j]，用 cse 提取公共子表达式
5. 用 codegen("F77") 或自定义模板输出格式：
   - 函数签名：void compute_hessian(const double* params, double* H_out)
   - H_out 按 row-major 存储
6. 在脚本末尾自动跑一遍数值验证：用 sympy.lambdify 计算几个测试点，与有限差分对比
```

**这就是 AI 在数学上无可替代的价值**——你描述意图，它写代码，机器跑出严谨结果。

## 第 6 周：符号 → 工业 C++

**周目标**：把 SymPy 输出变成项目里能编译的代码。

### 周一-周三：自动 CSE 与代码组织（6h）

**重要修正**：原版让你"手动让 AI 提取公共子表达式"——这是错的。**SymPy 自带 `cse()`**：

```python
from sympy import cse, ccode
common, reduced = cse(hessian_expressions)
for var, expr in common:
    print(f"const double {var} = {ccode(expr)};")
for expr in reduced:
    print(ccode(expr))
```

**这周的 prompt 不再是"让 AI 写 CSE"**，而是：
```
帮我写一段 SymPy 后处理脚本：
1. 输入：cse(...) 的输出
2. 把临时变量重命名为可读名字（基于其出现位置或依赖）
3. 输出格式：一个 C++ 函数，参数列表来自原始符号变量，
   函数体先声明所有公共子表达式（const double），再赋值给 H_out
4. 自动按依赖顺序排序临时变量声明
5. 输出到 std::cout 或写文件
```

### 周四-周日：Eigen 稀疏组装（4h）

模板：
```
我有 N 个顶点的网格，每个顶点对其邻域产生一个局部 Hessian（已通过 SymPy 生成）。
我需要把所有局部 Hessian 组装为全局稀疏矩阵 H ∈ R^{3N×3N}。

要求：
1. 用 Eigen::Triplet 收集所有非零项，最后一次性 setFromTriplets
2. 预先调用 reserve() 预估非零数（按每个顶点邻域大小估算）
3. 处理边界顶点（邻域元素少）
4. 对称性：只填上三角，最后用 selfadjointView<Eigen::Upper>() 引用
5. 给出完整代码 + 标注每一步的性能含义

不要使用 insert()——告诉 AI 为什么不能用（多次 insert 会触发多次重排）。
```

**学到的**：与 Eigen 配合时，**你要在 prompt 里主动否决错误方案**。AI 默认给的不一定是最优的。

### 周一-周三：浮点数 review 模板（6h）

```

1. **大数吃小数**：两个量级相差很大的数相加
2. **相近数相减**：catastrophic cancellation
```

# 附录 C：12 周的反模式（必看）

**这些事千万别做**：

1. ❌ **让 AI 帮你做架构决策**：什么数据结构、什么算法范式，AI 不懂你的项目权衡
2. ❌ **生成代码不读直接提交**：哪怕看着对，必须自己跑过测试用例
3. ❌ **在一个超长会话里反复迭代**：超过 30k tokens 后 AI 会变笨，**开新会话比纠缠强**
4. ❌ **让 AI 优化它刚写的代码**：它会强行找东西改，往往越改越差
5. ❌ **用 AI 处理超过 1000 行的单文件 refactor**：分小块，逐个推进
6. ❌ **完全依赖 AI 给的 OCCT/CGAL API 名**：必须去官方文档二次验证
7. ❌ **把生产数据贴进网页版**：API 不会用你数据训练（Anthropic 政策明确），但网页版有别的风险，敏感数据用 API 或本地工具

# 最后一句

这 12 周不是学 AI，**是用几何工程师的判断力去校准 AI**。

AI 永远会写出"看起来对、跑起来 90% 对、工业场景下 10% 翻车"的代码。**那 10% 就是你的护城河**。

整 12 周的目标只有一句话：
> **让能自动化的部分自动化，把你的精力释放到 AI 干不了的核心算法判断和容差设计上**。
